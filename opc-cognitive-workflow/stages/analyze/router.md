# 分析层路由规则

## 职责

理解阶段完成后，进入分析。根据当前问题的特征，判断**适合用什么框架**，提供选项给用户选择。

## 框架与问题的匹配

| 框架 | 最适合的问题类型 | 典型问题 |
|------|----------------|---------|
| **价值-成本矩阵** | 多个选项需要排优先级 | "该先做哪个？" |
| **四维评估** | 需要检验结论可靠性 | "这个判断靠谱吗？" |
| **要素分析** | 混沌问题需要结构化拆解 | "这个问题到底包含什么？" |
| **第一性原理** | 需要颠覆现有假设 | "为什么要这样做？" |
| **SWOT** | 竞争/战略分析 | "这个机会值不值得押？" |
| **横纵分析** | 时间线判断、跨期对比 | "未来会怎么走？" |

## 判断方式

1. 用户的问题本身：
   - 问"先做哪个/选哪个" → 价值-成本矩阵
   - 问"为什么/凭什么" → 四维评估或第一性原理
   - 问"是什么/包含什么" → 要素分析
   - 问"会怎样/趋势是什么" → 横纵分析
   - 问"要不要/值不值" → SWOT

2. 如果拿不准，问用户：
   > "你现在最缺的是判断依据（框架）、还是推翻现有假设的机会（第一性）？"

## 路由输出

用户选择框架后，调用 `frameworks/<framework-name>.md` 执行分析引导。

## 分析完成的标准

- 用户自己说出了结论（不是你给的）
- 用户自己得出了行动方案（不是照你说的做）
- 结论和框架之间有可追溯的逻辑链
