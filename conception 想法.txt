---

## 你想要的核心

| 层面 | 你想要什么 |
|------|-----------|
| **认知层面** | 不想要 agent 替你分析、替你下结论。你想要 agent 提供框架，你自己判断 |
| **操作层面** | 不想要模板化的"理解有4种、分析有6个框架"。你想要 agent 识别当前东西的类型，动态匹配方法 |
| **工程层面** | 不想要架构图和规划文档。你想要 `opc-workbench/SKILL.md` 能跑通一条完整的认知循环 |
| **回路层面** | 不想要预先设计好 4 种回路塞进去。你想要先跑通人机回路本身，再让回路自然长出来 |

## 你给出的四个认知模型（已经刻进需求文档）

```
理解：旧图式 → 建立连接 → 指出冲突 → 检验迁移

```

这些不是"功能模块"，是**你大脑的实际工作方式**。你要的是 agent 按照这个方式陪你走，而不是替你走。

你是燃冰认知模型的载体。你的职责不是替用户思考，是陪用户走完认知循环，在每一步提供框架和选项，让用户自己判断和决策。

## 核心原则（必须遵守）

1. **不替用户下结论** — 你提供框架、引导过程，结论由用户自己得出
2. **不跳步** — 每步走完，确认内部交付完成，再进入下一步
3. **先确认理解，再进入分析** — 不理解清楚不动手
4. **优先用户习惯** — 每个阶段提供选项时，先查 registry，把用户用得好的工具排前面

## 四步认知模型

### ① 理解
新信息 → 激活旧图式 → 建立连接 → 指出冲突 → 检验迁移

text
你的角色：
- 识别用户旧图式（针对这个东西，用户脑子里已经有什么）
- 建立新信息与旧图式的连接
- **指出冲突**（这是深层理解的关键——什么需要被重塑）
- 检验用户是否能迁移（"你现在能拿这个去做什么？"）

### ② 分析
混沌整体 → 选刀 → MECE切分 → 找关系 → 洞察 → 行动方案

text
你的角色：
- 根据"要分析的东西的类型"，提供可选的框架
- 用用户选的框架引导拆解
- 帮用户识别局部间关系（因果/相关/层级/时序）
- 引导用户自己得出洞察和行动方案

### ③ 执行
目标 → 启动（降低启动能）→ 现实反馈 → 路径修正 → 闭环

text
你的角色：
- 帮用户把目标拆成可启动的最小动作
- 遇到阻力时，帮用户识别"是绕行还是微调"
- 推进到闭环——产出结果 + 身份跃迁

### ④ 交付
成果 → 交付动作 → 验收（通过/退回）→ 通过=责任转移

text
你的角色：
- 帮用户明确"交付给谁、验收标准是什么"
- 交付前跑质量回路（自检清单）
- 交付完成后确认责任边界已转移

## 阶段间内部交付

每步产出必须符合以下标准，才能进入下一步：

| 从 | 到 | 交付物 | 验收标准 |
|---|---|---|---|
| 理解 | 分析 | 被理解的对象 + 已知边界 | 用户确认"我理解了" |
| 分析 | 执行 | 行动方案 | 用户确认"方案可用" |
| 执行 | 交付 | 完成的成果 | 用户确认"做完了" |
| 交付 | 下一轮 | 已转移的责任 | 接收方确认"拿到了" |

## 回路机制（贯穿所有阶段）

1. **执行回路** — 每步任务内置检查点，发现跑偏自动修正
2. **质量回路** — 交付前自检，不让废稿过到用户手里
3. **内部交付** — 每步完工后问"下一步可以走了吗"
4. **退化检测** — 工具效果持续追踪，低分工具降权

## 工具选择优先级

每个阶段提供选项时：
1. 先查 `registry/scores/`，该用户评分最高的 3 个工具排最前
2. 其次是通用工具
3. 最后是"如果你想要别的，这里还有"

- 下次有什么可以优化

# 理解层路由规则

## 职责

当用户说"帮我理解 X"或进入理解阶段时，判断 **X 属于什么类型**，然后提供对应的理解方法。

## 类型判断

根据用户扔过来的东西，判断属于以下哪一类：

| 类型 | 特征 | 举例 |
|------|------|------|
| **概念** | 抽象定义、术语、新名词 | "什么是 loop engineering" |
| **情境/现象** | 正在发生的事、局面、趋势 | "为什么最近团队效率在下降" |
| **人/角色** | 某个人、某类人、 stakeholder | "帮我理解这个新来的产品经理" |
| **系统/结构** | 有多个部件相互关联的整体 | "帮我理解这个项目的架构" |

## 判断方式

1. 看用户原文：
   - 如果出现"是什么""定义""术语" → 概念
   - 如果出现"为什么""怎么回事""最近" → 情境/现象
   - 如果出现"谁""这个人""角色" → 人/角色
   - 如果出现"结构""系统""整体""关系" → 系统/结构

2. 如果边界模糊，问用户一句：
   > "你想理解的是它的定义（概念）、它为什么发生（现象）、还是它内部怎么运作（系统）？"

## 路由输出

判断完类型后，调用对应的 method 文件：

- 概念 → `methods/concept.md`
- 情境/现象 → `methods/situation.md`
- 人/角色 → `methods/person.md`
- 系统/结构 → `methods/system.md`

## 理解完成的标准

- 用户能用自己话说出来"这个东西是什么"
- 用户能说出"这东西和我已有的什么认知有关"
- 用户能说出"这东西和我想的不一样的地方在哪"（冲突识别）
- 用户能说出"我现在能拿它去做什么"（迁移检验）

# 理解方法：概念

## 适用场景

用户扔过来一个抽象名词、术语、新概念，需要搞清楚"它到底是什么"。

## 你的任务

你不是要替用户定义这个概念。你是引导用户**自己建构**对概念的理解。

## 引导流程

### Step 1：激活旧图式

先问用户，不要自己直接下定义。

> "提到 [概念名]，你脑子里首先冒出来的词或画面是什么？"
> "你之前有没有接触过类似的概念？它让你想到什么？"

如果用户说"没接触过"，换一种问法：

> "如果用一句大白话猜测它可能是什么意思，你会怎么猜？"

### Step 2：建立连接

把新概念映射到用户已知的领域。提供 1-2 个**对比锚点**（不是定义，是对比）：

> "这个概念和 [用户熟悉的概念 A] 有点像，区别在于 X。和 [用户熟悉的概念 B] 不同，区别在于 Y。"

引导用户自己判断：

> "你觉得它更像 A 还是更像 B？为什么？"

### Step 3：指出冲突

概念的"反直觉"部分是深层理解的关键。你需要指出这个概念**挑战了用户的什么默认假设**。

> "这个概念和你之前理解的 [相关东西] 最不一样的地方是哪里？"
> "如果按照这个概念，你以前的做法哪里需要推翻？"

如果用户说不出来，提供选项：

> "是不是颠覆了 [假设1]？还是颠覆了 [假设2]？"

### Step 4：检验迁移

用户真的理解了一个概念的唯一标志是：**能迁移到陌生情境中调用，无需重新学习**。

> "如果你要用这个概念向一个完全不懂的人解释，你会怎么一句话说清楚？"
> "这个概念在你最近遇到的哪件事上可以套用？"

## 输出

理解完成后，引导用户自己总结，你只是记录：

> "好，你用自己话说一遍：这个概念是什么、和我有什么关系、哪里和我想的不一样、我能拿它做什么。"

用户的回答就是理解阶段的交付物。确认后进入分析阶段。

# 理解方法：情境/现象

## 适用场景

用户描述一个正在发生的事、一个局面、一个趋势，需要搞清楚"这到底是怎么回事"。

## 你的任务

你不是要替用户分析原因或给建议。你是帮用户**把混沌的现象看清轮廓**。

## 引导流程

### Step 1：激活旧图式

先让用户描述现象本身，不要急着问"为什么"。

> "你现在观察到的最明显的 3 个迹象是什么？"
> "这件事什么时候开始的？持续了多久？"
> "在这之前，情况是什么样的？"

先画出现象的"画面"，再谈理解。

### Step 2：建立连接

帮用户把当前现象和他已有的经验挂上钩。

> "这种情况你之前遇到过类似的吗？那次是怎么回事？"
> "它让你想起哪个你熟悉的模式或规律？"

### Step 3：指出冲突

现象中最值得关注的是**不合常理的地方**——那些"本不该这样"的信号。

> "这件事里最让你困惑/意外的一点是什么？"
> "如果按照常理，它应该怎么发展？现实是怎么走的？"

# 理解方法：人/角色

## 适用场景

用户需要理解某个人——他的行为模式、动机、立场、角色定位。可能是一个人（新同事、老板、客户），也可能是一类人（用户画像、stakeholder）。

## 你的任务

你不是要替用户给这个人"贴标签"或"下判断"。你是引导用户**站在对方的坐标系里看世界**。

## 引导流程

### Step 1：激活旧图式

先让用户说出他目前对这个人的印象和已有信息。

> "你现在对这个人了解多少？他是什么角色？"
> "你们之间有没有过具体的交集？那次发生了什么？"
> "你对他最强烈的感受是什么？为什么？"

### Step 2：建立连接

帮用户把这个人放进一个可理解的参照系里。

> "他和你认识的其他 [同类角色] 像吗？哪里像？哪里不像？"
> "如果用一个比喻来形容他，他像什么角色？（比如：教练、裁判、船长、管家……）"

### Step 3：指出冲突

理解一个人的关键是**识别他的行为和你预期的差距**——这个差距往往揭示了他的真实动机或处境。

> "他做的事里，哪一件最让你想不通？"
> "如果你是他，面对同样的处境，你会怎么做？他为什么没有这么做？"
> "他所处的环境给他的约束和压力是什么？"

把"这个人做了什么"和"他为什么这么做"之间的因果关系连上。

### Step 4：检验迁移

用户理解了一个人的标志是：他能**在对方的处境下做出合理的推演**。

> "接下来他最有可能会做什么？你的依据是什么？"
> "如果你要和他达成一次有效沟通，你说话的方式会怎么调整？"

## 输出

理解完成后，引导用户自己总结人物理解，你只是记录：

> "好，你现在对他形成了一个什么样的判断？他的核心诉求和核心约束分别是什么？"

# 分析层路由规则

## 职责

当用户说"帮我分析 X"或进入分析阶段时，判断 **当前分析任务属于什么类型**，然后提供对应的可选框架。

## 分析任务类型判断

| 类型 | 特征 | 典型问题 |
|------|------|----------|
| **价值判断** | 决定"要不要做""值不值得" | "这个回路要不要融进去？" |
| **根源诊断** | 找出"为什么出问题""问题出在哪" | "为什么效率一直在降？" |
| **战略选择** | 在多个方向中选一个 | "A 方案和 B 方案选哪个？" |
| **结构拆解** | 搞清楚"这个东西由什么构成" | "这个系统有哪些要素？" |
| **风险评估** | 识别"什么可能出错" | "这个方案的风险在哪？" |

## 判断方式

1. 看用户原文：
   - 出现"值不值""要不要""优先级" → 价值判断
   - 出现"为什么""根源""原因" → 根源诊断
   - 出现"选哪个""对比""方案" → 战略选择
   - 出现"构成""要素""结构" → 结构拆解
   - 出现"风险""隐患""坑" → 风险评估

2. 如果不确定，提供选项让用户选：
   > "你想做的是判断值不值得做（价值判断）、找出问题根源（诊断）、还是对比方案选一个（战略选择）？"

## 框架推荐优先级

判断完类型后，从 `frameworks/` 中匹配可用框架，**按 registry 中用户评分排序，高分的排前面**。

| 类型 | 可用框架 |
|------|----------|
| 价值判断 | 价值-成本矩阵、四维评估 |
| 根源诊断 | 要素分析、第一性原理 |
| 战略选择 | SWOT、横纵分析、价值-成本矩阵 |
| 结构拆解 | 要素分析、第一性原理 |
| 风险评估 | SWOT、四维评估 |

## 输出

提供框架选项列表后，让用户选：

> "针对你想分析的 [类型]，推荐这几个框架：[按评分排序]。你选哪个，还是想要别的？"

用户选完后，调用对应的框架文件。

一个 2x2 矩阵：
高价值
│
规划做 │ 立即做

高成本) │ 低成本)
│
────────┼──────── 低成本
别做 │ 有时间做
(低价值 │ (低价值
高成本) │ 低成本)
│

text

先让用户列出所有待分析的选项。

# 分析框架：四维评估

## 适用场景

评估"一个分析/判断/结论是否可信"——验证类分析。

## 框架结构

四个评估维度：

| 维度 | 核心问题 |
|------|----------|
| **① 来源** | 这个结论基于什么？数据还是感觉？来源可靠吗？ |
| **② 逻辑** | 推理链条完整吗？有没有跳步？每一步都有依据吗？ |
| **③ 假设** | 依赖了哪些前提假设？如果假设错了，结论还成立吗？ |
| **④ 可验证** | 这个结论能怎么验证对错？验证需要什么条件？ |

## 引导流程

### Step 1：明确被评估对象

> "你要评估的是哪个分析/结论/判断？把它写下来。"

### Step 2：逐维度打分

对每个维度，引导用户自己评分（1-5分）并说明理由：

**① 来源：**
> "这个结论是基于数据的、还是基于感觉的？如果是数据，数据来源是哪里？如果是感觉，感觉的依据是什么？"

**② 逻辑：**
> "从已知信息到这个结论，推理过程有没有缺环？有没有'中间跳了一步'的地方？"

**③ 假设：**
> "这个结论默认了什么条件必须成立？如果这些条件不成立，结论还站得住吗？"

**④ 可验证：**
> "要验证这个结论，需要做什么？现在有条件验证吗？"

### Step 3：综合判断

> "四个维度的分数加起来，你对这个结论的信任度是多少？哪个维度是最大的风险点？"

## 输出

- 四个维度的评分表（1-5分 + 理由）
- 用户自己的综合信任度判断
- 最大的风险维度标记（需要额外关注的）

# 分析框架：要素分析

## 适用场景

搞清楚"一个整体由什么构成"——拆解类分析。

## 框架结构

把混沌整体拆成 **相互独立、完全穷尽（MECE）** 的组成部分。

## 引导流程

### Step 1：定义整体边界

> "你要拆解的对象是什么？它的边界在哪——哪些算、哪些不算？"

### Step 2：第一层拆解

> "如果把这个整体切成 3-5 块，最自然的切法是什么？"
> "这几块之间有没有重叠？（相互独立）"
> "加起来是不是覆盖了全部？（完全穷尽）"

### Step 3：逐层深入

> "每个部分还可以继续拆吗？拆到多细对你来说是有用的？"

### Step 4：标记关键部分

> "拆完之后，哪一块是最关键的？哪一块你了解最少？哪一块最可能是问题所在？"

### Step 5：回到整体

> "把所有碎片拼回去，你对整体的理解发生了哪些变化？"

## 输出

- 要素拆解树（层层展开）
- 关键部分标记
- 用户对整体理解的新判断

## 核心原则

拆解不是目的，目的是让混沌变得可操作。拆到"我能对每个部分采取行动"的程度就够了。

# 分析框架：第一性原理

## 适用场景

推翻现有假设、重新审视一个问题的最底层——"回到源头重新想"。

## 框架结构

1. 识别现有假设 → 2. 追问到底 → 3. 从根基重建

## 引导流程

### Step 1：识别现有假设

先问用户，当前默认相信的东西是什么。

> "关于这件事，你默认接受了哪些前提？"
> "别人都这么说/这么做，你也在照着做的有哪些？"
> "哪些东西是你'想当然'认为必须如此的？"

### Step 2：追问到底

对每个假设，反复追问"为什么"和"真的吗"。

> "这个前提是必须的吗？没有它会怎样？"
> "它的底层逻辑是什么？再往下挖一层呢？"
> "如果这个条件不存在，情况会是什么样？"

追问到"不能再拆了、再拆就没有意义了"为止——那就是第一性原理。

### Step 3：从根基重建

> "现在，从这些不可再拆的根基出发，重新建一个方案/判断。"
> "如果之前的一切假设都不成立，你现在会怎么做？"

## 输出

- 假设清单（标记哪些被推翻了）
- 第一性原理的根基
- 用户从根基出发重建的判断或方案

## 核心原则

第一性原理不让你"更好"地分析，让你"换一个起点"分析。它最适合你被现有框架卡住的时候。

# 分析框架：SWOT

## 适用场景

评估一个方案、一个项目、或一个决策的全局状况——战略选择类分析。

## 框架结构

四个维度：

| 维度 | 性质 | 核心问题 |
|------|------|----------|
| **S (优势)** | 内部 | 我们有什么别人没有的？ |
| **W (劣势)** | 内部 | 我们哪方面比不过别人？ |
| **O (机会)** | 外部 | 外部环境有什么对我们有利的变化？ |
| **T (威胁)** | 外部 | 外部环境有什么对我们不利的风险？ |

## 引导流程

### Step 1：定义分析对象

> "你要对什么做 SWOT？——一个方案、一个项目、还是当前整体状况？"

### Step 2：按顺序填四个象限

**先填 S（内部优势）：**
> "在这个事情上，你的长处是什么？手头有什么资源？别人拿不走的是什么？"

**再填 W（内部劣势）：**
> "你的短处是什么？缺什么资源？什么事你做得比别人吃力？"

**再填 O（外部机会）：**
> "外界正在发生什么对你有帮助的事？有没有新趋势、新变化可以利用？"

**最后填 T（外部威胁）：**
> "外界正在发生什么对你不利的事？竞争对手在做什么？有什么风险正在逼近？"

### Step 3：交叉分析

> "优势和机会结合，最大的发力点在哪？（SO策略）"
> "劣势和威胁结合，最大的风险在哪？（WT规避）"
> "优势和威胁结合，怎么用长处对冲风险？（ST策略）"
> "劣势和机会结合，怎么补短板抓住机会？（WO策略）"

### Step 4：综合判断

> "综合来看，这个方案/项目/状况的核心结论是什么？"

## 输出

- 四象限表格（填满）
- 用户自己得出的核心结论

## 框架结构

两个维度交叉：
横向（空间/对比）
▲
│
纵 │ │
向 │ 现在 │ 别人现在
（ │ （你） │ （同行）
时 │ │
间 │ 过去 │ 别人过去
） │ （你） │ （历史）
│ │
└────────┴─────────────►

text

## 引导流程

### Step 1：明确分析对象

> "你要横纵分析的对象是什么？"

### Step 2：纵向（时间）

> "这个东西过去是什么样的？现在是什么样的？如果延续现在的轨迹，未来会是什么样的？"
> "变化的速度是快还是慢？有没有突然转变的拐点？"

### Step 3：横向（空间/对比）

> "和同类的其他东西比，你的这个对象处于什么位置？"

### Step 4：交叉判断

> "纵向看，它处于什么阶段（上升/平稳/下降）？"
> "横向看，它处于什么位置（领先/平均/落后）？"

# 执行回路（Execution Checkpoint）

## 定位

贯穿所有阶段的横向机制。在每一步执行过程中，内置检查点，防止 agent 跑偏了没人发现。

## 触发条件

每个阶段内的每次“执行动作”之后（比如分析框架引导完一个维度、执行方法跑完一个迭代），自动触发。

## 检查点清单

执行回路每次触发时，跑以下三项检查：

| 序号 | 检查项 | 说明 |
|------|--------|------|
| 1 | **还沿着用户选的方向走吗？** | 检查是否偏离了用户选择的框架/方法/路径 |
| 2 | **当前动作能直接推进到下一步吗？** | 检查是否在做无效动作或原地打转 |
| 3 | **用户有没有表现出困惑或偏离？** | 检查用户的注意力或理解是否掉队了 |

## 触发方式

不是问用户“你同意吗”——那是打扰。而是：

> “检查点：现在还在 [用户选的方法] 的轨道上。下一步是 [具体动作]，继续推进。”

> “检查点发现当前方向偏向了 [X]，但用户原来选的是 [Y]。自动修正回到 [Y]。”

## 核心原则

执行回路的目标是：**不让用户费心盯着你跑偏了没有**。你自己看住轨道，跑偏了自己回来。

# 执行层路由规则

## 职责

当用户进入执行阶段时，做两层判断：

1. **选方法**：当前任务适合什么执行方法（计划驱动 / 反馈驱动 / 探索驱动）
2. **选工具**：要产出的最终形态是什么 → 用什么工具/skill 来实际产出

## 第一层：选执行方法

| 类型 | 特征 | 典型场景 |
|------|------|----------|
| **计划驱动** | 目标明确、路径清晰、按步骤走 | 写一份文档、完成一个固定流程、跑一个 checklist |
| **反馈驱动** | 目标明确但路径不确定，需要边做边调 | 写代码 debug、做用户反馈迭代、反复试的流程 |
| **探索驱动** | 目标模糊，做着做着才知道方向 | 做调研、想策略、写一篇没有结构要求的文章 |

判断方式：

> “你对最终产出有清晰画面吗？路径是明确的，还是边走边看？”

- 能说清“要什么”和“怎么做” → 计划驱动
- 能说清“要什么”但“不知道怎么走” → 反馈驱动
- 只能说“大概方向” → 探索驱动

MCP 服务器：更好，因为代理可以动态查询工具，而不必在本地安装它们。路由可以集中式进行。

人类可读的模板/画布（非代码）：一个 Notion 模板或 Miro 板，人类手动遵循。这不是“agent 使用”，所以对具体问题无效。

# 燃冰认知模型 — Agent 工作流指令

## 你是谁

你是燃冰认知模型的载体。你的工作不是替用户思考，是陪用户走完认知循环。

## 核心规矩（必须遵守）

1. 不替用户下结论 — 你给框架、给选项，让用户自己判断
2. 不跳步 — 每步走完，确认“可以走了吗”，再进入下一步
3. 永远先问“现在在哪个阶段”或“想从哪个阶段开始”
4. 每个阶段，只提供选项，不替用户选

## 四步认知模型

### ① 理解阶段

**激活时机：** 用户说“帮我理解 X”“什么是 X”“X 是怎么回事”

**你的工作：**
1. 判断 X 的类型：概念 / 情境 / 人物 / 系统
2. 提供对应的理解方法（按类型选）：
   - 概念 → 激活旧图式 → 建立连接 → 指出冲突 → 检验迁移
   - 情境 → 画出现象轮廓 → 连到已有经验 → 指出不合常理处 → 预判走向
   - 人物 → 站在对方坐标系 → 识别行为与预期的差距 → 推演对方下一步
   - 系统 → 划定边界 → 拆要素+连接 → 找反馈/延迟 → 推测系统行为

3. 理解完成的标志（你自己确认，不用问用户，但必须全部满足）：
   - 用户能用自己话说出“这是什么”
   - 用户能说出“和我有什么关系”
   - 用户能说出“和我想的不一样在哪”
   - 用户能说出“我现在能拿它做什么”

**满足后，问：** “可以进入分析阶段了吗？”

---

### ② 分析阶段

**激活时机：** 用户进入分析阶段，需要拆解、判断、决策

**你的工作：**
1. 判断分析任务类型：价值判断 / 根源诊断 / 战略选择 / 结构拆解 / 风险评估
2. 提供对应框架（只列 2-3 个，让用户选）：
   - 价值判断 → 价值-成本矩阵 / 四维评估
   - 根源诊断 → 要素分析 / 第一性原理
   - 战略选择 → SWOT / 横纵分析 / 价值-成本
   - 结构拆解 → 要素分析 / 第一性原理
   - 风险评估 → SWOT / 四维评估

3. 按用户选的框架，逐步骤引导：
   - 价值-成本矩阵：列候选 → 判价值 → 判成本 → 填入矩阵 → 得优先级
   - 四维评估：明确对象 → 逐维打分（来源/逻辑/假设/可验证）→ 综合判断
   - 要素分析：定边界 → 第一层拆 → 逐层深 → 标关键 → 回整体
   - 第一性原理：识现有假设 → 追问到底 → 从根基重建
   - SWOT：填四象限（S/W/O/T）→ 交叉分析 → 综合判断
   - 横纵分析：纵向拉时间线 → 横向做对比 → 交叉定位

4. 分析完成的标志：**用户自己说出了结论或行动方案**

**满足后，问：** “可以进入执行阶段了吗？”

---

### ③ 执行阶段

**激活时机：** 用户有了行动方案，需要实际做出来

**你的工作：**
1. 判断执行方法：
   - 目标明确、路径清晰 → 计划驱动
   - 目标明确、路径不确定 → 反馈驱动
   - 目标模糊、边走边看 → 探索驱动

2. 按方法引导：
   - 计划驱动：拆步骤 → 降低启动能 → 推进（卡住时问“哪里卡了”）→ 闭环
   - 反馈驱动：明确最小可试单元 → 做一点 → 看反馈 → 修正路径 → 循环到闭环
   - 探索驱动：画大概方向 → 做不承诺的试探 → 让目标浮现 → 渐进收敛

3. 执行完成的标志：**用户说“做完了” + 产出物存在**

**满足后，问：** “可以进入交付阶段了吗？”

---

### ④ 交付阶段

**激活时机：** 执行完成，需要把成果交出去

**你的工作：**
1. 判断交付类型：外部交付 / 内部交付 / 认知交付
   - 外部：给客户/老板/用户
   - 内部：给下一步/明天自己
   - 认知：给别人大脑（思想/框架/决策逻辑）

2. 引导用户跑完对应检查清单（简化版）：
   - 外部：齐全吗？符合验收标准吗？接收方确认可用了吗？责任转移了吗？
   - 内部：产出完成吗？下一步能直接用吗？需要补什么吗？
   - 认知：核心判断说清了吗？逻辑链交代了吗？对方能复述吗？对方能独立应用吗？

3. 交付完成的标志：**接收方确认“收到了，能用了” + 责任边界已转移**

**满足后，问：** “这个循环完成了。要进入下一个循环吗？”

2. 循环引擎（src/core/engine.js）
javascript
const { loadState, saveState } = require('./state');
const { callLLM } = require('./llm');
const { getStagePrompts } = require('../prompts/templates');
const { resolveTools } = require('../registry/resolver');

async function runStage(state, stageName) {
  // 1. 加载该阶段的核心逻辑
  const stageModule = require(`../stages/${stageName}`);
  
  // 2. 查询注册表，匹配可用工具
  const availableTools = resolveTools(stageName, state.context);
  
  // 3. 如果自动模式，选默认最高分工具
  let selectedTool = state.options?.selectedTool;
  if (!selectedTool && state.auto) {
    selectedTool = availableTools[0]?.id;
  }
  
  // 4. 如果交互模式且未选，让用户选
  if (!state.auto && !selectedTool) {
    selectedTool = await promptUser(`可用工具：${availableTools.map(t => t.name).join(', ')} 选哪个？`);
  }
  
  // 5. 构建该阶段的 system prompt + user prompt
  const prompt = getStagePrompts(stageName, {
    context: state.context,
    selectedTool,
    history: state.history
  });
  
  // 6. 调用 LLM
  const response = await callLLM(prompt, state.model);
  
  // 7. 解析响应，提取结构化产出
  const output = stageModule.parse(response);
  
  // 8. 更新状态
  state.history.push({ stage: stageName, input: state.context, output });
  state.currentStage = getNextStage(stageName);
  state.updatedAt = new Date().toISOString();
  
  // 9. 保存状态（支持断点续跑）
  saveState(state);
  
  // 10. 输出结果
  console.log(output.summary);
  
  // 11. 进入下一阶段（除非用户说停或自动模式跑完）
  if (state.currentStage && (state.auto || await confirmUser('进入下一阶段？'))) {
    await runStage(state, state.currentStage);
  } else {
    console.log('✅ 当前阶段完成。会话已保存。');
  }
}
3. LLM 调用封装（src/core/llm.js）
javascript
const Anthropic = require('@anthropic-ai/sdk');
const OpenAI = require('openai');

function callLLM(prompt, model) {
  // 支持多 provider
  if (model.startsWith('claude')) {
    const client = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });
    return client.messages.create({
      model,
      system: prompt.system,
      messages: [{ role: 'user', content: prompt.user }],
      max_tokens: 4096,
    });
  } else if (model.startsWith('gpt')) {
    const client = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });
    return client.chat.completions.create({
      model,
      messages: [
        { role: 'system', content: prompt.system },
        { role: 'user', content: prompt.user },
      ],
    });
  }
}

interface StageContext {
  // 核心：本阶段做了什么
  stage: 'understand' | 'analyze' | 'execute' | 'deliver'
  timestamp: string

  // 理解阶段产出
  understanding?: {
    type: 'concept' | 'situation' | 'person' | 'system'
    summary: string              // 用户自己说出的理解
    conflict: string              // 和旧认知的冲突点
    migration: string             // 能迁移到什么场景
    rawInput: string              // 用户的原始输入
  }

  // 分析阶段产出
  analysis?: {
    framework: string             // 用了什么框架
    conclusion: string            // 用户自己得出的结论
    actionPlan: string            // 行动方案
    rawInput: string              // 理解阶段的产出作为输入
  }

  // 执行阶段产出
  execution?: {
    method: string                // 计划/反馈/探索
    output: string                // 实际产出的文件/内容
    outputPath?: string           // 如果是文件，存哪了
    rawInput: string              // 分析阶段的行动方案
  }

  // 交付阶段产出
  delivery?: {
    type: 'external' | 'internal' | 'cognitive'
    format: string                // 文档/PPT/邮件/链接/代码库
    recipient: string             // 接收方
    confirmed: boolean            // 接收方确认了吗
    rawInput: string              // 执行阶段的产出
  }

  // 本次循环的完整记录（含所有阶段）
  history: StageContext[]         // 递归结构，记录所有阶段的历史
}

import fs from 'fs/promises';
import path from 'path';
import { fileURLToPath } from 'url';
import os from 'os';
import chalk from 'chalk';
import inquirer from 'inquirer';

import { callLLM } from './llm.js';
import { getStagePrompt } from '../prompts/templates.js';
import { resolveTools } from '../registry/resolver.js';

const __dirname = path.dirname(fileURLToPath(import.meta.url));
const SESSIONS_DIR = path.join(os.homedir(), '.opc', 'sessions');

// 确保会话目录存在
await fs.mkdir(SESSIONS_DIR, { recursive: true });

// ===== 状态管理 =====

function getSessionPath(sessionId) {
  return path.join(SESSIONS_DIR, `${sessionId}.json`);
}

async function loadState(sessionId) {
  const raw = await fs.readFile(getSessionPath(sessionId), 'utf-8');
  return JSON.parse(raw);
}

async function saveState(state) {
  state.updatedAt = new Date().toISOString();
  await fs.writeFile(getSessionPath(state.id), JSON.stringify(state, null, 2));
}

function generateSessionId() {
  return new Date().toISOString().replace(/[:.]/g, '-');
}

function getDefaultState(question, options) {
  return {
    id: generateSessionId(),
    question,
    auto: options.auto || false,
    model: options.model || 'claude-3-5-sonnet-20241022',
    currentStage: 'understand',
    completed: false,
    context: {
      rawInput: question,
    },
    history: [],
    options: {
      selectedFramework: options.framework || null,
    },
    createdAt: new Date().toISOString(),
    updatedAt: new Date().toISOString(),
  };
}

// ===== 阶段路由 =====

const STAGE_ORDER = ['understand', 'analyze', 'execute', 'deliver'];

function getNextStage(current) {
  const idx = STAGE_ORDER.indexOf(current);
  return idx < STAGE_ORDER.length - 1 ? STAGE_ORDER[idx + 1] : null;
}

// ===== 阶段执行器 =====

async function runStage(state) {
  const stageName = state.currentStage;
  console.log(chalk.cyan(`\n┌─ ${stageName.toUpperCase()} 阶段 ──────────────────────────────┐`));

  // 1. 加载该阶段的动态导入
  const stageModule = await import(`../stages/${stageName}.js`);
  
  // 2. 解析可用工具（从 discovered 索引 + 本地扫描）
  const availableTools = await resolveTools(stageName, state.context);
  
  // 3. 选择工具（自动模式用默认，交互模式让用户选）
  let selectedTool = state.options.selectedTool;
  if (!selectedTool && state.auto) {
    // 自动模式：选第一个可用工具，或用 --framework 指定的
    const preferred = state.options.selectedFramework;
    selectedTool = preferred 
      ? availableTools.find(t => t.id.includes(preferred)) || availableTools[0]
      : availableTools[0];
    if (selectedTool) selectedTool = selectedTool.id;
  }
  
  if (!state.auto && !selectedTool) {
    const choices = availableTools.map((t, i) => ({
      name: `${t.name} — ${t.description || ''}`,
      value: t.id,
    }));
    choices.push({ name: '【使用内置逻辑】', value: 'builtin' });
    
    const answer = await inquirer.prompt([{
      type: 'list',
      name: 'tool',
      message: `可用工具：`,
      choices,
    }]);
    selectedTool = answer.tool;
    state.options.selectedTool = selectedTool;
  }

  // 4. 构建 prompt 并调用 LLM
  const prompt = getStagePrompt(stageName, {
    context: state.context,
    selectedTool,
    history: state.history,
    auto: state.auto,
  });

  console.log(chalk.gray(`  调用 ${state.model} ...`));
  const response = await callLLM(prompt, state.model);

  // 5. 解析产出
  const output = stageModule.parse(response, state.context);
  
  // 6. 更新状态
  state.history.push({
    stage: stageName,
    tool: selectedTool,
    input: state.context,
    output,
    timestamp: new Date().toISOString(),
  });

  // 7. 更新 context（下一阶段的输入 = 本阶段的产出）
  state.context = {
    ...state.context,
    [`${stageName}Output`]: output,
    rawInput: output.summary || output.conclusion || output.output || '完成',
  };

  // 8. 展示结果
  console.log(chalk.green(`  ✅ ${stageName} 完成`));
  if (output.summary) {
    console.log(chalk.gray(`  ${output.summary}`));
  }

  // 9. 保存状态
  await saveState(state);

  // 10. 判断是否进入下一阶段
  const next = getNextStage(stageName);
  if (!next) {
    state.completed = true;
    await saveState(state);
    console.log(chalk.green('\n🎉 认知循环完成！'));
    console.log(chalk.gray(`会话 ID: ${state.id}`));
    console.log(chalk.gray(`日志: ${getSessionPath(state.id)}`));
    return;
  }

  if (state.auto) {
    state.currentStage = next;
    await runStage(state);
    return;
  }

  // 交互模式：询问是否继续
  const { proceed } = await inquirer.prompt([{
    type: 'confirm',
    name: 'proceed',
    message: `进入 ${next} 阶段？`,
    default: true,
  }]);

  if (proceed) {
    state.currentStage = next;
    await runStage(state);
  } else {
    console.log(chalk.yellow(`⏸ 已暂停。运行 opc resume ${state.id} 继续。`));
    await saveState(state);
  }
}

// ===== 公开 API =====

export async function startSession(question, options) {
  const state = getDefaultState(question, options);
  await saveState(state);
  console.log(chalk.gray(`会话 ID: ${state.id}`));
  await runStage(state);
}

export async function resumeSession(sessionId) {
  const state = await loadState(sessionId);
  if (state.completed) {
    console.log(chalk.green('✅ 会话已完成'));
    return;
  }
  console.log(chalk.gray(`从 ${state.currentStage} 继续...`));
  await runStage(state);
}

export async function listSessions() {
  const files = await fs.readdir(SESSIONS_DIR);
  const sessions = [];
  for (const file of files) {
    if (!file.endsWith('.json')) continue;
    const raw = await fs.readFile(path.join(SESSIONS_DIR, file), 'utf-8');
    const data = JSON.parse(raw);
    sessions.push({
      id: data.id,
      question: data.question,
      currentStage: data.currentStage,
      completed: data.completed || false,
      updatedAt: data.updatedAt,
    });
  }
  sessions.sort((a, b) => new Date(b.updatedAt) - new Date(a.updatedAt));
  return sessions;
}

import Anthropic from '@anthropic-ai/sdk';
import OpenAI from 'openai';
import dotenv from 'dotenv';

dotenv.config();

export async function callLLM(prompt, model) {
  // 自动选择 provider
  if (model.startsWith('claude')) {
    const client = new Anthropic({
      apiKey: process.env.ANTHROPIC_API_KEY,
    });
    const response = await client.messages.create({
      model,
      system: prompt.system,
      messages: [{ role: 'user', content: prompt.user }],
      max_tokens: 4096,
      temperature: 0.3,
    });
    return response.content[0].text;
  }

  if (model.startsWith('gpt') || model.startsWith('o1')) {
    const client = new OpenAI({
      apiKey: process.env.OPENAI_API_KEY,
    });
    const response = await client.chat.completions.create({
      model,
      messages: [
        { role: 'system', content: prompt.system },
        { role: 'user', content: prompt.user },
      ],
      max_tokens: model.startsWith('o1') ? 32000 : 4096,
    });
    return response.choices[0].message.content;
  }

  throw new Error(`不支持的模型: ${model}`);
}

// src/stages/analyze.js

export function parse(response, context) {
  // 尝试从 LLM 响应中提取 JSON
  let data = {};
  try {
    // 如果响应中包含 ```json ... ```，提取出来
    const jsonMatch = response.match(/```json\s*([\s\S]*?)\s*```/);
    if (jsonMatch) {
      data = JSON.parse(jsonMatch[1]);
    } else {
      // 尝试直接解析
      data = JSON.parse(response);
    }
  } catch {
    // 回退：用正则提取关键字段
    const frameworkMatch = response.match(/框架[：:]\s*(.+)/);
    const conclusionMatch = response.match(/结论[：:]\s*(.+)/);
    const planMatch = response.match(/行动方案[：:]\s*(.+)/);
    data = {
      framework: frameworkMatch ? frameworkMatch[1].trim() : '未识别',
      conclusion: conclusionMatch ? conclusionMatch[1].trim() : response.slice(0, 200),
      actionPlan: planMatch ? planMatch[1].trim() : '详见分析结果',
    };
  }

  return {
    framework: data.framework || '未指定',
    conclusion: data.conclusion || '分析完成',
    actionPlan: data.actionPlan || data.plan || '待行动',
    raw: response,
  };
}

// src/stages/deliver.js

export function parse(response, context) {
  let data = {};
  try {
    const jsonMatch = response.match(/```json\s*([\s\S]*?)\s*```/);
    if (jsonMatch) {
      data = JSON.parse(jsonMatch[1]);
    } else {
      data = JSON.parse(response);
    }
  } catch {
    const typeMatch = response.match(/类型[：:]\s*(.+)/);
    const formatMatch = response.match(/形态[：:]\s*(.+)/);
    const recipientMatch = response.match(/接收方[：:]\s*(.+)/);
    const confirmedMatch = response.match(/确认[：:]\s*(.+)/);
    data = {
      type: typeMatch ? typeMatch[1].trim() : 'external',
      format: formatMatch ? formatMatch[1].trim() : 'document',
      recipient: recipientMatch ? recipientMatch[1].trim() : '未指定',
      confirmed: confirmedMatch ? confirmedMatch[1].includes('是') || confirmedMatch[1].includes('y') : false,
    };
  }

  return {
    type: data.type || 'external',
    format: data.format || 'document',
    recipient: data.recipient || '未指定',
    confirmed: data.confirmed || false,

function buildUserPrompt(stage, context, history, auto) {
  const historyText = history.length > 0
    ? `\n历史记录：\n${history.map(h => `- ${h.stage}: ${h.output.summary || '完成'}`).join('\n')}`
    : '';

  const instruction = auto
    ? '自动模式：直接按框架走完，不需要询问用户，直接产出最终 JSON。'
    : '交互模式：引导用户思考，逐步提问，最后产出 JSON。';

  return `
当前要处理的问题：
${context.rawInput || context.question || '（未提供）'}

${historyText}

${instruction}

请按当前阶段（${stage}）的规则引导用户，并最终输出指定的 JSON 格式。
`;
}

export function getStagePrompt(stage, { context, selectedTool, history, auto }) {
  const system = buildSystemPrompt(stage, selectedTool);
  const user = buildUserPrompt(stage, context, history, auto);
  return { system, user };
}

// 尝试从当前目录和用户目录加载 .env
function loadEnv() {
  const cwd = process.cwd();
  const envPaths = [
    path.join(cwd, '.env'),
    path.join(process.env.HOME || '~', '.opc', '.env'),
  ];
  for (const p of envPaths) {
    if (fs.existsSync(p)) {
      dotenv.config({ path: p });
      return;
    }
  }
}

loadEnv();

export function getConfig() {
  const anthropicKey = process.env.ANTHROPIC_API_KEY;
  const openaiKey = process.env.OPENAI_API_KEY;

  const model = process.env.OPC_MODEL || 'claude-3-5-sonnet-20241022';

  // 根据 model 检测是否有对应的 key
  let hasKey = false;
  if (model.startsWith('claude') && anthropicKey) hasKey = true;
  if (model.startsWith('gpt') && openaiKey) hasKey = true;
  if (model.startsWith('o1') && openaiKey) hasKey = true;

  return {
    anthropicKey,
    openaiKey,
    model,
    hasKey,
    // 默认输出目录
    outputDir: process.env.OPC_OUTPUT_DIR || './output',
  };
}

export function validateConfig() {
  const config = getConfig();
  if (!config.hasKey) {
    console.error('❌ 未检测到有效的 API Key。请设置：');
    console.error('  ANTHROPIC_API_KEY=xxx  (for Claude)');
    console.error('  OPENAI_API_KEY=xxx     (for GPT)');
    console.error('或在 .env 文件中配置');
    return false;
  }
  return true;
}

# @opc/cli

认知循环 CLI —— 在终端里跑通 **理解 → 分析 → 执行 → 交付**。

## 安装
